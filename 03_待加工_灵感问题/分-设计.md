# 待办解决过程（设计先例）（2026-01-21）
1. 从下→上的经验积累到一定程度后→**基本能力搭建好**
2. 过去的能力无法满足新的需求，萌生出从上到下的想法→**发现问题+动机**
	- 周复盘→从上到下规划萌芽
3. 尝试新方法+Excel（ai）→ **尝试+实践+解决问题**
	- 从上到下的被动设计（根据现实需求设计）→固定模版/流程
		1. 待办（实践）
			1. 方案1：收集待办，设计Excel表格，填写（待办）
			2. 方案2：看板页面（标题+内容）+属性（Excel→Obsidian表格）→不联动→pass
				- 新增
					- 看板页面新增
					- 表格中新增
				- 处理
					- 看不见内容，相同模块待办无法同时整合
				- 删除
					- 看板页面已完成→归档
					- 表格中要删除
			3. 方案3：将Excel表格字段转为待办页面属性，将属性和对应的待办联动起来（标题+内容+属性）
				1. 一个待办的所有内容都联动起来，通过内容+属性
				2. 通过图表满足不同需求的呈现，比Obsidian页面中的表格更直观（只要记录了数据，图表可以随意变化，从不同维度）
				3. 其他可以结构化设计
					- 主动性（属性）
						- 主动
							- 每日日记（事后属性）
								- 时长、状态，工作内容，日期
								- 习惯：体重，打字速度，起床时间
							- 周/月复盘日记：除了一些日期属性，status（这个是后加的）
							- 待办：之前没有
							- 灵感：只有一个Type属性
							- ai对话：到现在都没有管理
							- 播客：作者属性
							- 综上所述，我对Obsidian中的属性认识很浅薄，要不没有属性，纯粹当做笔记记录，最多的，也不过是每日日记，记录追踪一些习惯数据和一天中的宏观数据而已；直到这次待办页面的属性设计，让Obsidian中的笔记从内容载体→过程管控，实现了更深层次的需求
								- 管控建立在有目标的前提下，这样才和现实有差距，才需要进行管控追踪，如果连目标都没有，那就是记录到什么程度就是什么程度
						- 被动
							- 书
								- 豆瓣：页数，评论数，类别
									- 虽然有这些字段，但我挑选的时候好像还是根据喜好
								- 微信：控制进度，开始结束日期
									- 虽然有这些字段，但更多还是作为一种事后记录，而非事中控制
		2. 周复盘（实践）
			- 设计周复盘模版
				- 计划
				- 执行偏差
				- 小试牛刀
				- 复盘
	- 从上到下的主动规划（吸收新的理论，主动设计而非被动设计）
		1. 书（提取理论→实践）
		2. 模型（理论→实践，融入到系统中）
		- 为什么要加入这两个，是因为虽然有了从上到下管控的想法，但如果只等事情发生了，再来设计，那终究还是从下到上的思想的抽取；但如果每日注入新的理论，那就是从预防，提前规划的角度
4. 然后通过工具（Obsidian中属性记录+Component结构呈现）实现管控→**系统设计+反应程序**
	- 待办系统设计
		- **新建待办**：新增创建待办按钮组件，将灵感直接捕捉为标题，捕捉+新建同时完成
		- **查询待办**：让Ai生成dataviewjs代码，实现自定义开始+结束日期，可以将灵感文件夹下该日期范围内打卡完成的待办提取出来 
		- **待办笔记的处理**：从待办文件夹→灵感文件，提取推理过程+解决方案
		- **模块汇总**：修改NoteRefactor插件的规则，实现灵感文件名（哪个模块）→一级标题（问题）→行为推进（解决方案），纵向汇总模块解决维度
	- 复盘模版系统设计
		- 把年度/月度目标集成到周模版中，周模版中增加本周目标
		- 把周目标集成到日模版中，实现层层递进
		- 增加倒计时策略
			- 周复盘日通过remainder提醒，上午复盘、下午安排待办
5. 大量练习反馈→**形成习惯**
	- 模型
		- 设置模型打卡时间（和待办打卡绑定），增加打卡仪表盘
	- 待办
		- 重启复盘（放松）仪表盘，从复盘打卡→待办打卡
		- 增加早上从清单中取待办打卡
		- 增加晚上执行偏差填写打卡
6. **记录→微调至成功**
7. 精进能力

# 习惯追踪表页面设计（2026-01-24）
- 其实我觉得所谓的结构化设计，本质上是模版设计，形成模版1.0
- 表格→页面→属性/标题下内容
	- 表格
		- 表格是最终呈现，所有习惯所有要素的汇总
		- 实现预览功能，提供宏观视角
	- 页面→习惯
		- 单个习惯所有要素汇总
		- 一个模版将一个习惯的所有要素固定下来
			- 对于有问题的习惯快速发现问题
			- 对于新习惯可以快速拆解，培养
	- 属性/标题下内容→元素
		- 习惯的微观层面
		- 拆解习惯元素，通过具体解决每个元素的问题，形成习惯的解决方案，才能形成最后的习惯，聚沙成塔
- 为什么要重构习惯追踪（从单页面表格填写→多页面属性+内容管理）
	- 习惯可以解构成单个要素：启动信号+时间分配+调整+反应程序等
		- 针对每个具体的习惯，拆解→发现问题→提出解决方案→调整→迭代固化，目的是要解决问题，而非责备自己
		- 通过成功培养出的习惯，指导正在调整的新习惯和变动中的习惯，将成功经验扩大化
	- 页面不利于记录，追踪，维护数据，无法指导现实工作，失去表格意义

		|                           |           单页面           | 多页面属性+内容管理  |
		|:-------------------------:|:--------------------------:|:--------------------:|
		| **维护成本（时间+设计）** |             低             |      高（设计）      |
		|         **功能**          | 快速拆解元素，但维护不方便 | 集成到系统，方便追踪 |
		|         **频率**          |           一次性           |         频繁         |
	- 对当下的习惯来说，表格填写也可以解决问题，但结构化设计可以
		1. 减轻旧习惯的维护成本，快速调整
		2. 灵活配置图表，最大化发掘数据信息
		3. 新习惯可以快速拆解，稳定下来，而不是周末复盘的时候发现问题，解决问题，反馈比较长，且粒度比较粗，和具体执行有差别（实时+灵活）
		4. 实现精细化控制
			- 一个习惯一个页面
			- 甚至针对单个习惯的每次调整，都是一个页面
	- 待办作为前例，打了样，提供了成功案例
